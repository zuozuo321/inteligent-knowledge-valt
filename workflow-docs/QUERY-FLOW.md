# 查询与回填流程 (Query & Backfill Flow)

## 概述

查询流程是用户从知识库获取信息的路径，回填流程则是将 AI 在回答过程中产生的新知识保存回知识库的机制。

这两个流程共同构成了知识库的"读-写"循环，确保知识库不断增长和优化。

## 查询流程

### 流程示意图

```
用户提问
    |
    v
[1. 理解问题]
    | AI 分析问题类型和关键概念
    v
[2. 查索引层]
    | 搜索 index/ 定位相关主题
    |
    +-- 找到索引? --+
    |               |
    是               否
    |               |
    v               v
[3. 读 wiki 层]   [4. 搜索全库]
    | 精读相关     | 搜索 wiki/ 和 raw/
    | wiki/ 文件   |
    |               |
    +-- 找到答案? -+
    |               |
    是               否
    |               |
    v               v
[5. 合成回答]     [6. 告知缺口]
    | 整理信息     | 说明知识库中
    | 生成回答     | 缺少相关信息
    v               |
[7. 输出回答]      |
    |               v
    v          [8. 搜索外部]
[是否回填?]     | WebFetch 或
    |           | 其他来源
    |           |
    是          否
    |           |
    v           v
回填流程      结束
```

### 详细步骤

#### 步骤 1: 理解问题

AI 收到用户问题后，首先分析：
- 问题涉及的主题领域
- 问题类型（概念解释、操作指导、比较分析等）
- 关键概念和术语
- 回答的预期深度和格式

#### 步骤 2: 查索引层

```
AI 调用: search_notes("关键词", folder="index")
```

- 在 `$VAULT_PATH/index/` 中搜索关键词
- 如果找到匹配的索引文件，读取索引获取主题概览和相关链接
- 索引文件会列出该主题下的核心概念，每个概念都有指向 wiki/ 的链接

#### 步骤 3: 精读知识层

```
AI 调用: read_note("wiki/topic/concept")
```

- 根据索引中的链接，读取 wiki/ 中的具体文件
- 深入理解概念的定义、原理、示例
- 注意文件中的交叉引用，可能指向相关概念
- 如果答案分散在多个文件中，综合理解后合成回答

#### 步骤 4: 搜索全库（索引未命中时）

```
AI 调用: search_notes("关键词")   # 搜索整个知识库
```

- 如果索引层没有相关内容，在整个知识库中搜索
- 包括 wiki/ 和 raw/ 层
- 如果找到相关 raw/ 素材，说明该素材尚未编译

#### 步骤 5: 合成回答

AI 将查找到的信息整理为回答：

**回答格式**：
1. **直接答案**：简洁地回答用户问题
2. **知识库引用**：标注信息来源的路径
3. **补充说明**：相关概念的交叉引用
4. **知识缺口**：如果知识库信息不完整，指出缺口

**示例**：
```
根据知识库中的信息，自注意力机制的核心公式是：

    Attention(Q, K, V) = softmax(QK^T / sqrt(d_k)) V

来源：[[wiki/deep-learning/attention-mechanism]]
相关概念：[[wiki/deep-learning/transformer-architecture]]

补充说明：该机制的关键创新在于...

注意：知识库中关于"稀疏注意力"的信息尚未覆盖，
如果需要，我可以搜索相关素材后补充。
```

#### 步骤 6: 告知知识缺口

如果知识库中没有足够信息：
- 明确告知用户缺少哪些信息
- 建议是否需要从外部获取（B站视频、网页等）
- 或建议是否需要从 raw/ 中编译未处理的素材

### 查询深度分级

| 级别 | 查询范围 | 适用场景 |
|------|---------|---------|
| 浅层 | 仅 index/ | 快速了解主题是否存在 |
| 中层 | index/ + wiki/ | 获取概念解释 |
| 深层 | index/ + wiki/ + raw/ + 外部 | 深入研究 |
| 全面 | 全库 + 外部 + 重新编译 | 复杂问题 |

## 回填流程

### 触发条件

回填在以下情况触发：
1. AI 在回答中生成了新的、有价值的信息
2. AI 综合多个 wiki/ 文件形成了新的见解
3. 用户明确要求保存某段对话内容
4. 发现了现有知识库中的信息缺失

### 回填目标

| 内容类型 | 目标位置 | 说明 |
|---------|---------|------|
| 新概念 | wiki/ | 创建新文件 |
| 补充信息 | wiki/ (现有文件) | 追加到已有概念 |
| 索引更新 | index/ | 添加新链接 |
| 分析报告 | outputs/ | 保存完整对话 |
| 原始素材 | raw/ | 直接保存未处理内容 |

### 回填格式

#### 补充到 wiki/
```markdown
<!-- AI 补充内容 -->
<!-- 补充于 2025-06-16，来源：用户问答 -->

### 补充：实际应用场景

在实际工程中，[概念]通常用于...

---
> 参考：[[outputs/2025-06-16-qa-session]]
```

#### 创建新 wiki/ 文件
按照 [VAULT-SCHEMA.md](VAULT-SCHEMA.md) 中的规范创建完整文件。

#### 保存对话记录
```
$VAULT_PATH/outputs/YYYY-MM-DD-<topic>.md
```

### 回填验证

回填完成后，AI 应验证：
1. 新内容是否正确定位
2. 是否有正确的 frontmatter 和引用
3. 是否需要更新索引
4. 内容是否与现有知识一致（不重复、不矛盾）

## 循环优化

知识库的查询和回填形成持续优化的循环：

```
查询 -> 发现缺口 -> 回填 -> 知识库增长
  ^                            |
  +----------------------------+
```

每次查询都可能触发回填，每次回填都让后续查询更准确。
这种循环是 Karpathy 方法论的核心优势。

### 优化指标

- **命中率**：查询时能在 wiki/ 中找到答案的比例
- **回填率**：有价值的回答内容被回填的比例
- **编译延迟**：素材从 raw/ 到 wiki/ 的平均时间
- **索引覆盖率**：wiki/ 文件被 index/ 引用的比例

## 注意事项

1. **优先使用索引**：不要跳过 index/ 直接搜索 wiki/，索引能提供更好的上下文
2. **保持引用链条**：回填内容必须标注来源，方便追踪
3. **避免重复**：回填前检查是否已有相似内容
4. **标记不确定性**：如果回填内容来自外部且未验证，使用 `status/needs-review` 标签
